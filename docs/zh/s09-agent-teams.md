# s09: Team Mailboxes

## 为什么需要这一章

光增加更多 agent 并不够。只有当消息、身份和路由都显式化后，协作才真正可教、可调试。

## 核心机制

- 给 worker 稳定名称
- 通过 mailbox 路由通信
- 让每个 worker 拥有自己的循环和状态
- 把消息传递当成协作表面

## 这一章如何映射到 Python 代码

`agents/s09_agent_teams.py` 引入 mailbox bus 和 team manager，而不是假装所有 worker 共享一个隐藏大脑。

## 这里刻意简化了什么

消息传输层故意保持朴素，教学目标是显式协作，而不是传输层炫技。

## 建议你自己试一试

- 先打开 `agents/` 里本章对应的 Python 文件，观察新增类、工具或运行时状态。
- 给模型一个必须用到本章机制的任务，而不是只问概念定义。
- 关注“状态是存在哪儿、谁负责更新、父子/前后台/多 worker 如何看到它”。

## 它如何连接到下一章

当消息面建立之后，运行时就可以开始区分“普通对话”和“类型化协调请求”。
